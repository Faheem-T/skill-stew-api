# Ordering in Distributed Systems

Before implementing the notification system, I researched how to approach ordering in distributed systems — specifically for messaging and notification delivery where order matters. This research also informed decisions in other parts of the system.

## The Core Requirements

For ordered data in a distributed system, IDs need to be:

- **Time-ordered** — newer items should have larger IDs for efficient sorting
- **Distributed** — generated independently across multiple servers without coordination
- **Collision-free** — no duplicates, even under high throughput
- **Millisecond precision** (ideally) — second-level precision may not be enough for high-frequency data

## Approaches Evaluated

### Timestamps

Using the insertion timestamp as the ordering key.

**Why it fails:** Clock skew across servers means there's no single source of "correct" time. Even with microsecond precision, collisions are possible, and indexes cannot tolerate collisions on unique keys.

---

### MongoDB ObjectIDs

Letting MongoDB handle ordering with its built-in 96-bit ObjectID.

**Trade-off:** ObjectIDs are sortable and work in distributed systems, but they only have **second-level precision**. Multiple documents created within the same second may not sort correctly. For messaging systems with rapid-fire messages, this is insufficient — but for lower-frequency data like notifications, it can be acceptable.

---

### Auto-incrementing Primary Keys

Sequential keys generated by the database.

**Why it fails at scale:** Works with a single DB instance, but breaks with horizontal scaling. Multiple shards either need a central authority to decide the next ID (creates a bottleneck and single point of failure) or generate IDs independently (causes collisions). The entire write path becomes centralized.

---

### UUID v4

128-bit randomly generated identifiers.

**Why it fails for ordering:** UUIDs v4 are completely random — no time component. They work well for unique identification but are terrible for database indexing (random inserts fragment B-tree indexes) and cannot be sorted by creation time.

---

### ULID (Universally Unique Lexicographically Sortable Identifier)

128-bit, time-ordered, distributed. Proposed in 2016 to address UUID v4's poor ordering. Conceptually identical to UUID v7, with a more URL-friendly encoding (26 characters vs 36).

**Verdict:** Viable, but UUID v7 is preferred due to standardization.

---

### UUID v7

128-bit, time-ordered (millisecond precision), distributed, standardized in 2024. The industry recognized the need for time-sortable UUIDs and formalized what ULID had proposed.

**Verdict:** The default choice for most use cases. No server configuration needed, standard library support, millisecond precision.

---

### Twitter's Snowflake IDs

64-bit IDs: 41 bits timestamp + 10 bits machine ID + sequence bits. Introduced by Twitter in 2010, before UUIDs were standardized for time-ordering.

**Trade-offs:**

- Half the size of UUIDs (64-bit vs 128-bit) — significant at scale for index size
- Requires machine ID management (Twitter used Zookeeper for dynamic assignment)
- Clock skew is a real problem since 41 bits hold the timestamp — requires NTP synchronization
- At small scale, machine IDs can be hardcoded and cloud providers handle clock sync

**Verdict:** Only worth the operational complexity at very high scale (100M+ records) where index size becomes a bottleneck.

## Comparison

| Method         | Size      | Time-ordered | Distributed | Collision-free | Complexity |
| -------------- | --------- | ------------ | ----------- | -------------- | ---------- |
| Timestamps     | Varies    | Yes          | No          | No             | Low        |
| MongoDB OID    | 96-bit    | Yes (1s)     | Yes         | Yes            | Low        |
| Auto-increment | 32/64-bit | Yes          | No          | Yes            | Medium     |
| UUID v4        | 128-bit   | No           | Yes         | Yes            | Low        |
| ULID           | 128-bit   | Yes          | Yes         | Yes            | Low        |
| UUID v7        | 128-bit   | Yes          | Yes         | Yes            | Low        |
| Snowflake      | 64-bit    | Yes          | Yes         | Yes            | Medium     |

## What I Chose

- **UUID v7** for most of the system — used across all services for entity IDs (`uuidv7()`). Standardized, millisecond precision, no coordination needed.
- **MongoDB ObjectID** for notification ordering — second-level precision is sufficient for notifications, and it avoids adding a separate ID column since MongoDB generates it automatically. Notifications are sorted by `_id` in descending order.

## What I Learned

- The research was prompted by planning the notification system, but the findings applied broadly. Understanding _why_ UUID v4 is bad for indexing and _why_ auto-increment breaks in distributed systems made me more deliberate about ID choices across the entire project.
- There's no single "best" ID strategy — the right choice depends on precision requirements, scale, and operational complexity you're willing to accept.
- For the vast majority of applications, UUID v7 is the right default. You only need to consider alternatives like Snowflake when operating at extreme scale where 64-bit vs 128-bit index sizes matter.
